# 암호화폐 시그널 대시보드 재구축 설계

## 1. 목적

기존 단일 HTML 기반 시그널 스캐너를 무료 GitHub Pages 정적 웹앱으로 재구축한다.
참고 이미지의 넓은 금융 대시보드 구성을 반영하고, Bybit 중심의 직접 추가 분석,
자동 후보 스캔, 시장 국면 분석, 과거 데이터 백테스트를 제공한다.

완성된 앱은 별도 서버, 데이터베이스, 유료 서비스, API 키 없이 사용할 수 있어야
한다. 브라우저에서 거래소 공개 API만 호출한다.

## 2. 확인된 기존 문제

기존 `index.html`은 약 298KB의 단일 파일이며 화면, API 호출, 분석, 저장, 알림
로직이 한 파일에 섞여 있다.

직접 추가한 종목도 매매 모드를 선택하지 않으면 숨겨지며, 모드를 선택한 뒤에도
스타일, 신뢰도, 진입 가능 플랜 조건을 통과해야만 카드가 표시된다. 이 구조 때문에
HBAR처럼 검색 후 추가한 종목이 실제로는 목록에 들어갔더라도 사용자는 탐지되지
않는 것처럼 보게 된다.

기존의 적중률 기능은 진짜 백테스트가 아니다. 신호가 발생한 뒤 브라우저 저장소에
거래를 등록하고, 이후 현재 가격이 TP 또는 SL에 닿았는지 추적하는 실시간 기록
기능이다. 이 기능은 제거하고 과거 캔들 기반 백테스트로 교체한다.

## 3. 범위

### 3.1 핵심 기능

- 직접 추가 종목 분석
- Bybit 전체 종목 자동 후보 스캔
- 시장 국면 분석
- 선택 종목 과거 데이터 백테스트
- 백테스트 거래 내역 CSV 다운로드
- 사용자 선택 기반 로컬 설정 저장
- 상세 오류 진단

### 3.2 보조 기능

기존 섹터 자금 흐름, 추천 목록, 알림, 트레이드 플랜 등은 검토 후 유효한 기능만
접이식 보조 패널로 유지한다. 핵심 기능의 안정성을 해치는 부가 기능은 포함하지
않는다.

### 3.3 거래소 지원

- Bybit: 직접 추가 분석, 자동 후보 스캔, 시장 국면, 백테스트
- Binance: 직접 추가 분석만 지원

## 4. 파일 구조

```text
index.html
css/
  styles.css
js/
  app.js
  api/
    bybit.js
    binance.js
  analysis/
    indicators.js
    signals.js
    market-regime.js
    trade-plan.js
  backtest/
    engine.js
    metrics.js
    csv.js
  ui/
    dashboard.js
    manual-assets.js
    scanner.js
    backtest-view.js
    diagnostics.js

```

모듈은 역할별로 분리한다. API 모듈은 원격 데이터 조회와 오류 분류만 담당한다.
분석 모듈은 순수 함수 중심으로 구성한다. UI 모듈은 계산 로직을 직접 포함하지
않는다.

## 5. 화면 구성

참고 이미지처럼 짙은 네이비 계열의 넓은 데스크톱 대시보드를 기본으로 한다.
모바일에서는 패널이 한 열로 재배치되며 기능은 줄이지 않는다.

### 5.1 상단 영역

- 앱 이름
- 기본 거래소 Bybit 표시
- 공개 API 상태
- 마지막 갱신 시간
- 설정 버튼

### 5.2 요약 카드

- 시장 국면: 상승, 중립, 하락
- 직접 추가 종목 수
- 자동 스캔 후보 수
- 최근 백테스트 핵심 결과

### 5.3 본문 탭

- `직접 추가`: 사용자가 검색해서 추가한 종목을 항상 표시한다.
- `자동 후보`: Bybit 자동 스캔 결과를 모드별로 필터링한다.
- `백테스트`: 종목, 기간, 비용, 대기 캔들 수를 설정하고 결과를 확인한다.
- `부가 분석`: 유지하기로 결정한 기존 부가 기능을 접이식 패널로 제공한다.

## 6. 직접 추가 종목 정책

직접 추가 종목은 자동 후보 목록과 분리한다. HBAR처럼 현재 선택한 매매 모드의
조건을 충족하지 못한 종목도 카드 자체는 항상 표시한다.

카드는 다음 정보를 보여준다.

- 데이터 조회 상태
- 현재 분석 결과
- 모드별 적합 여부
- 제외된 경우 그 이유
- 일부 지표가 없을 때 분석 가능한 범위
- 펼쳐볼 수 있는 상세 진단 정보

종목 입력은 대문자로 정규화하고 `USDT` 접미사는 내부에서 일관되게 처리한다.
허용 문자 이외의 값은 요청 전에 거부한다.

## 7. 백테스트 설계

### 7.1 데이터

Bybit 과거 캔들 공개 API를 사용한다. 사용자가 직접 추가한 여러 종목 중 원하는
종목을 선택하여 실행한다. 종목별 결과와 합산 결과를 모두 제공한다.

기간은 빠른 선택 버튼 `30일`, `90일`, `180일`, `365일`과 시작일, 종료일 직접
입력을 모두 제공한다.

### 7.2 비교 결과

다음 결과를 분리하여 보여준다.

- 공통 확정 신호 성과
- 스캘핑 모드 성과
- 단타 모드 성과
- 데일리 모드 성과
- 스윙 모드 성과

### 7.3 미래 데이터 참조 방지

신호는 해당 캔들이 닫힌 뒤에만 확정한다. 신호 계산 시점 이후의 캔들 값은 신호
판정에 사용할 수 없다.

신호 확정 직후 즉시 체결하지 않는다. 이후 캔들에서 지정 진입 구간에 실제로
닿은 경우만 체결한다. 대기 기간 안에 닿지 않으면 미체결로 기록한다.

기본 대기 기간은 다음과 같으며 설정에서 수정할 수 있다.

- 스캘핑: `6개` 캔들
- 단타: `12개` 캔들
- 데일리: `6개` 캔들
- 스윙: `4개` 캔들

### 7.4 비용과 체결

Bybit 비용을 기본값으로 적용하고 사용자가 수정할 수 있게 한다.

- 왕복 수수료: `0.11%`
- 왕복 슬리피지: `0.20%`

동일 캔들 안에서 TP와 SL이 모두 도달하면 보수적으로 SL을 먼저 적용한다.

### 7.5 성과 지표

- 거래 수
- 승률
- 누적 수익률
- 최대 낙폭
- 평균 손익비
- 기대값
- Profit Factor
- 평균 보유 기간
- 최대 연속 손실
- 미체결 수
- 종목별 비교
- 모드별 비교

거래별 진입, 청산, 비용, 손익 내역은 CSV로 다운로드할 수 있다.

## 8. 오류 처리

직접 추가 종목은 오류가 발생해도 카드가 사라지지 않는다. 오류를 다음과 같이
분류한다.

- 거래소에 종목 없음
- 일부 보조지표 미제공
- API 호출 제한
- 네트워크 또는 CORS 오류
- 응답 형식 오류

카드에는 사용자용 원인과 해결 방법을 먼저 표시한다. 상세 진단을 펼치면 거래소,
요청 종류, 오류 코드, 발생 시각을 확인할 수 있다.

## 9. 저장 정책

기본적으로 API 키와 개인정보를 다루지 않는다. 설정에서 사용자가 저장을 허용한
경우에만 브라우저 `localStorage`를 사용한다.

저장 가능한 항목은 다음으로 제한한다.

- 직접 추가 종목 목록
- UI 설정
- 백테스트 입력 기본값

API 키, 개인정보, 원격 API 원문 응답은 저장하지 않는다.

## 10. 보안 기준

- 공개 API만 호출한다.
- API 키 입력 UI를 만들지 않는다.
- 외부 스크립트 의존성을 최소화한다.
- 사용자 입력 종목명을 허용 문자 기준으로 검증한다.
- 외부 API 데이터를 HTML 문자열로 직접 삽입하지 않고 안전하게 렌더링한다.
- 요청 실패와 응답 형식 오류를 명시적으로 처리한다.
- 로컬 저장은 사용자 동의가 있을 때만 활성화한다.
- GitHub Pages에 필요한 정적 파일만 배포한다.

## 11. 검증 기준

완성 전에 다음을 검증한다.

- HBAR를 검색하고 직접 추가하면 항상 직접 추가 탭에 카드가 보인다.
- HBAR가 모드 조건을 충족하지 않아도 제외 이유를 확인할 수 있다.
- Bybit 자동 후보 스캔이 직접 추가 목록과 분리되어 동작한다.
- Binance는 직접 추가 분석만 제공한다.
- 빠른 기간 버튼과 직접 날짜 입력으로 백테스트를 실행할 수 있다.
- 공통 확정 신호와 모드별 성과가 분리되어 표시된다.
- 동일 캔들 TP/SL 동시 도달은 SL로 처리된다.
- 진입 구간 미도달 거래는 미체결로 기록된다.
- 수수료와 슬리피지가 결과에 반영되고 수정 가능하다.
- 상세 지표와 CSV 다운로드가 동작한다.
- 저장을 거부하면 새로고침 후 사용자 설정이 복원되지 않는다.
- 저장을 허용하면 허용된 항목만 복원된다.
- 데스크톱과 모바일 레이아웃을 확인한다.
- 보안 검토 결과를 사용자에게 보고한다.
- GitHub 저장소를 만들고 업로드하여 GitHub Pages 배포가 가능하도록 한다.

## 12. 배포

구현과 검증이 끝나면 로컬 Git 저장소에 커밋한다. GitHub 로그인 상태를 확인한 뒤
사용자에게 저장소 이름과 공개 여부를 질문한다. 승인받은 설정으로 GitHub 저장소를
생성하고 업로드한다. GitHub Pages 활성화가 필요하면 저장소 설정을 적용하고 배포
URL을 확인한다.
